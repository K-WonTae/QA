# 학사행정 Q&A 도우미 — 개선 작업 지시서 (Claude Code용)

> 대상: 대학교 학사행정 문서 기반 Q&A 도우미 (FastAPI + Claude **CLI**)
> 전제: **`claude` CLI subprocess 구조를 유지한다. Anthropic SDK로 전환하지 않는다.**
> 목표: 토큰/비용·지연 절감 + 잠재적 교착·프로세스 누수 제거 + 코드 정리

---

## 0. 작업 규칙 (반드시 준수)

1. **각 작업은 대상 파일을 먼저 읽고 현재 동작을 확인한 뒤 수정한다.** 라인 번호는 참고용이므로 추측으로 고치지 말 것.
2. **다음 보안 불변 조건을 절대 훼손하지 않는다:**
   - `security.py` 화이트리스트 단일 관문 (root 밖·심볼릭링크·확장자 외 차단)
   - 모든 SQL은 파라미터 바인딩 (f-string/format 조립 금지)
   - 프롬프트 인젝션 격리 (`<<<DOCUMENT>>>` / `<<<QUESTION>>>` + 토큰 제거)
   - CLI 실행: shell 미사용, `--tools ""`, 격리 cwd, 최소 env, 타임아웃 강제
3. **한 작업 = 한 커밋.** 커밋 메시지에 작업 ID(예: `A-2`)를 포함한다.
4. **각 변경 후 SSE 스트리밍이 정상 동작하는지** 직접 1회 질의로 확인한다.
5. **"결정 필요 항목"(§4)은 구현 전에 사용자에게 값을 먼저 물어본다.** 임의로 정하지 말 것.

## 1. CLI 전제하의 비용 최적화 전략 (먼저 이해할 것)

캐싱이 불가능하므로 비용은 아래 세 축으로만 줄인다.

- **호출 수 ↓**: `@규정`의 2회 호출을 1회로 (A-1). CLI는 매 호출마다 node 프로세스를 새로 띄우므로, 호출 1회를 줄이면 토큰뿐 아니라 프로세스 생성 지연도 함께 준다.
- **컨텍스트 크기 ↓**: 전역 파일 캡(A-2) + 상한 하향(A-3) + 출처별 예산 공정 분배(A-4) + 헤더 바이트 반영(B-4).
- **모델 분리**: 규정 선택 같은 가벼운 작업은 `--model`로 경량 모델(Haiku 계열)을 쓰고, 본 답변만 상위 모델을 쓴다.

> **A-6(프롬프트 캐싱)은 이번 범위에서 제외.** CLI `-p` 경로로는 `cache_control` 적용이 불가능하다. 캐싱이 필요해지면 그때 SDK 전환을 별도 과제로 검토한다.

## 2. 작업 목록 (우선순위 순)

### P0 — 계측 먼저 (이후 모든 최적화의 효과 측정용)

#### A-5. 실토큰 usage 계측
- **위치**: `app/claude_runner.py` (stream-json 파싱부, 분석상 :193 부근), `persist_ask` / DB 스키마
- **현재**: stream-json의 usage 정보를 무시. 토큰은 추정만 가능.
- **작업**:
  1. `--output-format stream-json` 출력 1회를 **원본 그대로 캡처**해 이벤트 스키마를 먼저 확인한다 (`message_start`의 `usage.input_tokens`/캐시 토큰, 최종 `result` 이벤트의 `usage`·누적 output 등 — 설치된 CLI 버전마다 다를 수 있으니 실제 출력으로 확정).
  2. 스트림 파서에서 `input_tokens`, `output_tokens`(누적 최종값), `cache_read_input_tokens`, `cache_creation_input_tokens`를 추출한다.
  3. DB에 컬럼 추가(`input_tokens, output_tokens, cache_read, cache_write, model`) + 마이그레이션. `persist_ask`에서 기록.
- **완료 기준**: 질의 1건마다 DB에 실토큰이 남고, 운영 로그(메타)에도 input/output 토큰이 찍힌다.

### P1 — 비용/지연 즉시 절감

#### A-1. `@규정` 2회 호출 제거/축소 + 경량 모델
- **위치**: `app/pdf_source.py` (`select_regulations_llm`, 분석상 :277), `select_regulations`
- **현재**: 항상 LLM 선택 먼저 → 실패 시 키워드 폴백. 이 호출이 `prepare_ask` 안(스트림 열기 전)에서 블로킹 → 첫 토큰 지연 큼.
- **작업** (§4에서 정책 확정 후):
  1. **순서 역전**: 키워드/제목 매칭(`select_regulations`)을 **먼저** 수행한다.
  2. 후보가 **1개로 좁혀지면 LLM 선택 호출을 생략**한다.
  3. 후보가 모호할 때만 LLM 선택을 호출하되, **`--model`로 경량 모델**을 지정한다 (설치된 CLI가 받는 정확한 모델 별칭/ID를 먼저 확인할 것).
  4. (선택) 제목-키워드 정적 인덱스를 보강해 키워드 단계의 적중률을 높인다.
- **완료 기준**: 단일 규정이 명확한 질의에서 선택 LLM 호출 0회. 선택 호출이 일어나는 경우 경량 모델로 수행되어 첫 토큰 지연이 줄어든다.

#### A-2. 전역 문서 수 상한 추가
- **위치**: `app/router.py` `_gather_with_links` (분석상 :107)
- **현재**: `MAX_FILES_PER_QUERY=8`이 **출처별** 상한이라 연계출처가 있으면 전역 누적(로그상 21개).
- **작업**: `_gather_with_links`에 **전역 누적 카운터**를 두고 합산 파일 수가 전역 캡(§4에서 값 확정)에 도달하면 더 이상 파일을 추가하지 않는다. 주출처를 우선 채운다.
- **완료 기준**: 연계출처 포함 질의에서도 전역 파일 수가 캡을 넘지 않는다.

#### A-3. 컨텍스트 상한 하향 + 환경변수화
- **위치**: `app/config.py` `MAX_TOTAL_CONTEXT_BYTES` (분석상 :49)
- **현재**: 200KB 고정 → 질의당 5만~7만 토큰.
- **작업**: 값을 환경변수로 읽도록 바꾸고, 기본값을 보수적으로 낮춘다(§4에서 목표값 확정). 한글은 토큰이 추정보다 높게 나올 수 있으므로 A-5 계측으로 낮춘 뒤 실토큰을 재확인.
- **완료 기준**: 환경변수로 상한 조절 가능. 낮춘 값에서 답변 품질이 유지되는지 샘플 질의로 확인.

#### A-4. 연계출처 예산 공정 분배
- **위치**: `app/router.py` (분석상 :105-118)
- **현재**: 첫 출처가 상한을 다 쓰면 연계출처는 0바이트 기여인데도 **스캔은 수행**됨(낭비).
- **작업**: 출처별 **최소 보장 예산** 또는 균등 분배 로직 추가. **배정 예산이 0이 될 출처는 스캔 자체를 건너뛴다**(디스크 작업 절감).
- **완료 기준**: 연계출처도 최소한의 컨텍스트를 확보하거나, 기여 불가 시 스캔을 생략한다.

#### B-4. 헤더 바이트를 예산에 반영
- **위치**: `app/router.py` (분석상 :113-116)
- **현재**: `remaining`에서 `docs`만 빼고 `header` 길이는 미반영 → 미세하게 예산 초과 가능.
- **작업**: `remaining` 계산 시 `header` 길이도 차감.
- **완료 기준**: 실제 전송 바이트가 상한을 넘지 않는다.

### P2 — 안정성 (CLI 유지하므로 필수)

#### B-2. 스트리밍 중 stderr 드레인 (교착 방지)
- **위치**: `app/claude_runner.py` `run_claude_stream` (분석상 :184-217)
- **현재**: stdout만 읽고 stderr는 루프 종료 후 읽음 → CLI가 stderr로 많이 쓰면 파이프 버퍼가 차서 블로킹 → 90초 타임아웃까지 멈춤.
- **작업**: **별도 데몬 스레드**로 stderr를 지속적으로 버퍼에 드레인하고, 프로세스 종료 후 join한다.
- **완료 기준**: stderr 출력이 많은 상황에서도 스트리밍이 멈추지 않는다.

#### B-3. 비스트림 `run_claude` 타임아웃 시 자식 트리 정리
- **위치**: `app/claude_runner.py` (분석상 :104, `_terminate_tree`)
- **현재**: 스트림판은 `_terminate_tree`로 처리하지만 비스트림 `run_claude`(= `@규정` 선택 호출)는 미처리 → Windows에서 손자(node) 좀비 가능.
- **작업**: 비스트림 경로의 `TimeoutExpired`에서도 **기존 `_terminate_tree`를 재사용**해 프로세스 트리를 정리한다.
- **완료 기준**: 선택 호출 타임아웃 후 잔존 node 프로세스가 없다.
- **참고**: A-1에서 선택 LLM 호출을 제거하기로 결정되면 이 경로의 위험은 자연 감소하나, 비스트림 경로가 남아 있는 한 수정은 유지한다.

#### D-3. `_minimal_env`에 Windows 설정 디렉터리 추가
- **위치**: `app/claude_runner.py` (분석상 :60), `app/config.py` (분석상 :59)
- **현재**: `APPDATA`/`LOCALAPPDATA` 부재 → 환경 변화 시 CLI가 설정 디렉터리를 못 찾을 소지.
- **작업**: 최소 env 원칙은 유지하되 Windows에서 `APPDATA`, `LOCALAPPDATA`를 화이트리스트로 추가한다.
- **완료 기준**: 깨끗한 환경에서도 CLI가 정상 기동.

### P3 — 정리 / 유지보수

#### B-1. 죽은 분류기 코드 제거
- **위치**: `app/prompt_builder.py` (분석상 :41-58, `CLASSIFY_RULE`/`build_classify_prompt`)
- **작업**: 전역 참조가 없음을 grep으로 확인한 뒤 **제거**한다. (`category`는 `src["label"]`로 충분하며, 분류기를 연결하면 호출 1회가 추가되어 비용↑이므로 연결하지 않는다.)
- **완료 기준**: 미사용 코드 제거, 기능 회귀 없음.

#### C-1. 콘솔 로그 한글 깨짐 수정
- **위치**: `app/logging_conf.py` (분석상 :49-51)
- **현재**: 파일핸들러는 utf-8이나 콘솔 `StreamHandler`가 Windows cp949로 모지바케.
- **작업**: 콘솔 핸들러 인코딩을 utf-8로 지정(또는 `sys.stdout.reconfigure(encoding="utf-8")` / `PYTHONUTF8=1`). **파일 핸들러는 건드리지 않는다.**
- **완료 기준**: 콘솔에 한글이 정상 출력.

#### C-2. 중복 엔드포인트 공용화
- **위치**: `/api/ask`(`handle_ask`)와 `/api/ask/stream`; 프론트는 stream만 사용(`static/index.html` :647)
- **작업**: 공통 로직(검증·라우팅·문서수집)을 헬퍼로 추출해 두 경로가 공유하게 한다. (§4에서 비스트림 엔드포인트 **삭제 여부** 확정.)
- **완료 기준**: 로직 중복 제거, stream 경로 정상.

#### C-3. 진단 잔재 파일 정리
- **위치**: `data/_diag.py`, `data/_verify_links.py`, `data/_diag_out.txt` 등
- **작업**: data 폴더에서 분리(별도 `tools/` 이동 또는 삭제). data 폴더에는 운영 데이터만 남긴다.
- **완료 기준**: data 폴더에 진단 스크립트/산출물이 없다.

#### C-4. CSRF 토큰 재시작 무효화 (결정 필요)
- **위치**: `app/main.py` (분석상 :47)
- **현재**: 서버 재시작 시 토큰 변경 → 열린 페이지의 POST가 403.
- **작업**: §4 결정에 따라 (a) 시크릿을 환경변수/파일로 고정해 재시작에도 유지하거나, (b) 의도된 동작이면 그대로 둔다.

### P4 — 품질 / 성능

#### D-1. 스캔 400개 초과 시 조용한 품질 저하에 신호 추가
- **위치**: `app/security.py` (분석상 :179)
- **현재**: 텍스트 파일 >400이면 내용 스캔을 끄고 파일명 점수만 사용하는데 사용자/로그에 신호가 없음.
- **작업**: 이 모드로 전환될 때 **경고 로그**를 남기고, `dev_view`가 켜진 응답에는 "파일명 기반 축소 검색이 적용됨" 신호를 포함한다.
- **완료 기준**: 대형 폴더 질의 시 축소 검색 적용 사실이 드러난다.

#### D-2. 매 질의 전체 파일 재읽기 → mtime 캐시
- **위치**: `app/security.py` `select_context` (분석상 :186-222)
- **현재**: 질의마다 모든 텍스트 파일을 디스크에서 읽어 점수화.
- **작업**: 파일 경로+mtime을 키로 한 **인메모리 캐시**를 도입해 변경되지 않은 파일은 재읽기를 생략한다. (캐시 무효화는 mtime 변경 기준.)
- **완료 기준**: 동일 폴더 반복 질의 시 디스크 읽기/지연 감소, 결과 동일.

## 3. 권장 실행 순서

1. **A-5**(계측) — 먼저 넣어 이후 효과를 숫자로 검증.
2. **A-2 → A-3 → B-4 → A-4**(컨텍스트 축소) — 독립적이고 위험 낮음, 비용 즉시 절감.
3. **A-1**(호출 축소 + 경량 모델) — 정책 확정 후.
4. **B-2 → B-3 → D-3**(안정성) — CLI 유지하므로 필수.
5. **B-1 → C-1 → C-2 → C-3**(정리).
6. **D-1 → D-2**(품질/성능).
7. **C-4**(결정에 따라).

각 단계마다 A-5의 실토큰 로그로 before/after를 비교 기록할 것.

## 4. 구현 전 사용자에게 확정받을 항목

아래는 임의로 정하지 말고 **먼저 질문**한다.

1. **`@규정` LLM 선택 정책**: (a) 완전 제거(키워드만) / (b) 키워드로 못 좁힐 때만 LLM / (c) 현행 유지 — 어느 것?
2. **`MAX_TOTAL_CONTEXT_BYTES` 목표값**: 현 200KB → 목표(예: 96KB)?
3. **전역 파일 캡 값**: 총 몇 개(예: 8~10)?
4. **비스트림 엔드포인트(`/api/ask`)**: 삭제 vs 공용화 후 유지?
5. **CSRF 토큰 영속화**: 시크릿 고정(재시작에도 유지) vs 현행 유지?
