# 학사행정 Q&A 도우미 — 후속 작업 지시서 (Claude Code용)

> 범위: ① 출처 표기(citation) · ② 평가셋(golden Q&A) · ③ 피드백 버튼(👍/👎)
> 전제: 기존과 동일 — **`claude` CLI subprocess 구조 유지, SDK 전환하지 않음.**

---

## 0. 작업 규칙 (이전 지시서와 동일하게 준수)

1. **대상 파일을 먼저 읽고 현재 동작을 확인한 뒤** 수정한다. 추측 금지.
2. **보안 불변 조건을 절대 훼손하지 않는다:** `security.py` 화이트리스트 단일 관문, 모든 SQL 파라미터 바인딩, 프롬프트 인젝션 격리(`<<<DOCUMENT>>>`/`<<<QUESTION>>>`/`<<<CATALOG>>>` + `_sanitize`), shell 미사용·`--tools ""`·격리 cwd·최소 env·타임아웃.
3. **새로 추가되는 모든 사용자 입력(피드백 코멘트 등)은 파라미터 바인딩 + 길이 제한**하고, **절대 프롬프트로 되먹이지 않는다.**
4. 한 작업 = 한 커밋. 커밋 메시지에 작업 ID 포함.
5. 변경 후 두 경로(`@테이블` 로컬, `@규정`) SSE(meta→delta→done) 정상 동작 확인.

## 0-1. 미리 정한 기본값 (바꿀 항목만 사용자에게 확인)

- 출처 표기: **인라인(답변 말미) + 결정적 출처 목록을 SSE `done` 이벤트로 전달 + UI "참고 문서" 패널** 모두 적용. 모델이 댄 출처는 실제 제공 출처 집합과 대조해 **목록에 없는 출처는 제거**.
- 평가셋: 위치 `tests/eval/golden.jsonl`, 기본은 **결정적 검사**(의도·출처·포함어·금지어), LLM 심판(`--judge`, haiku)은 **기본 꺼짐**.
- 피드백: **별도 `feedback` 테이블**(`message_id` FK), 👎 시 **선택 코멘트(≤500자)**, 👎 답변을 **추려보는 export 스크립트** 제공.

## 공통 선행 작업 (①②③ 모두 의존) — 먼저 구현

### S-0. SSE `done` 이벤트에 `message_id`·`sources` 노출
- **위치**: `app/router.py`(SSE 전송부), `persist_ask`, `static/index.html`(SSE 수신부)
- **현재**: A-5로 `chat_messages`에 답변 행(토큰 포함)이 저장되지만, 그 **행 id와 실제 사용된 출처 목록**이 클라이언트로 안 나감.
- **작업**:
  1. 컨텍스트에 실제 포함된 출처 문서 목록을 `_gather_with_links`/`select_context` 단계에서 수집해 보관한다(경로·표시명).
  2. 답변 저장 시 그 `message_id`를 확보하고, SSE `done` 이벤트 payload에 `{ message_id, sources: [{path, title}] }`를 추가한다.
  3. 프론트는 `done`에서 이 둘을 받아 보관(②·③에서 사용).
- **완료 기준**: `done` 이벤트에 `message_id`와 실제 사용된 출처 목록이 실린다. A 즉답 경로에서도 동일하게 채워진다.

## ① 출처 표기 (Citation)

### F-1. 답변 인라인 출처 + 결정적 출처 목록
- **위치**: `app/prompt_builder.py`(시스템 규칙), `app/router.py`(done 이벤트 — S-0), `static/index.html`(렌더)
- **현재**: doc 블록에 출처 헤더는 있으나, 답변이 어느 문서를 근거로 했는지 사용자가 알 수 없음.
- **작업**:
  1. **시스템 규칙 추가**: "답변은 제공된 `<<<DOCUMENT>>>` 출처 헤더에 근거해야 하며, 답변 말미에 사용한 근거 문서명을 명시하라. **제공된 문서에 근거가 없으면 지어내지 말고 '제공된 문서에서 근거를 찾지 못함'이라고 답하라**(단, '존재하지 않는다'고 단정하지는 말 것)."
     - ※ 기존 "없다고 단정 마라" 규칙과 모순되지 않게 위 문장으로 **통합**한다.
  2. **닫힌 집합 제시**: 프롬프트에 이번 질의에서 인용 가능한 출처명 목록을 명시해, 모델이 그 안에서만 고르게 한다(환각 출처 방지).
  3. **사후 검증**: 모델이 답변에 적은 출처명을 S-0의 실제 출처 집합과 대조하여, **집합에 없는 출처는 답변에서 제거/표시**한다.
  4. **UI**: 답변 아래 "참고 문서" 섹션을 S-0의 `sources`로 렌더(모델 자기보고가 아니라 **실제 포함 출처**가 신뢰 기준).
  5. **A 즉답 경로**(`_build_inventory_answer`)도 동일하게 출처(집계에 쓴 index.md들)를 표시.
- **완료 기준**: Claude 답변 말미에 근거 문서가 표기되고, 그 출처는 실제 제공 집합을 벗어나지 않는다. UI에 "참고 문서" 패널이 뜬다. 근거 없을 때 환각 대신 "근거 못 찾음"으로 답한다.
- **검증 케이스**: "장학세부분류 코드값" → 답변에 해당 출처 표기 + 패널 노출. 근거 없는 질의 → "근거 못 찾음".

## ② 평가셋 (Golden Q&A 회귀 하네스)

### F-2. 평가 데이터셋 + 러너
- **위치**: 신규 `tests/eval/golden.jsonl`, `tests/eval/run.py`
- **현재**: 단위 테스트는 라우팅·가드 로직만 검증, 답변 품질 회귀를 못 잡음.
- **작업**:
  1. **데이터셋**(`golden.jsonl`) 케이스 스키마:
     ```json
     {"id":"tbl_list","question":"등록된 테이블 종류를 간단히 알려줘",
      "expect_intent":"inventory",
      "must_include":["테이블"], "must_not":["테이블 없음","존재하지 않"],
      "must_cite":["학사/테이블/index.md","행정/테이블/index.md"]}
     ```
     필드: `expect_intent`(inventory|content), `must_include`, `must_not`, `must_cite`(부분 일치 허용).
  2. **시드 케이스(필수 포함)**:
     - 원래 버그 회귀: 위 `tbl_list` — `must_not`에 "테이블 없음/존재하지 않" 고정.
     - `장학세부분류 코드값` → content, `must_include`=["H0201","성적우수","106"].
     - `@규정` 1건, `@지식` 1건, 후속질문 1건, A 오탐 방지 1건(특정 문서명 포함 질의).
  3. **러너**(`python -m tests.eval.run`): 각 케이스에 대해 **실제 답변 파이프라인**(prepare_ask→run_claude_stream, A 즉답 포함)을 **격리 cwd·읽기전용**으로 실행해 답변·출처·의도·토큰을 수집하고 검사.
     - 결정적 검사(기본): 의도 일치, `must_include`/`must_not` 부분일치, `must_cite` 출처 포함.
     - LLM 심판(`--judge`, **기본 꺼짐**): haiku로 답변 vs 기준 의미 일치 채점.
  4. **리포트**: 케이스별 pass/fail + 토큰 + 추정 비용(`input + 0.1×cache_read + 1.25×cache_write`, A-5 컬럼 사용)을 표로 출력하고 `tests/eval/report_<timestamp>.md`로 저장.
  5. **운영 데이터 격리**: 평가는 운영 DB를 건드리지 않는다(임시/별도 컨텍스트).
- **완료 기준**: 시드 케이스가 모두 통과하고, 리포트에 총 토큰·추정 비용이 찍힌다. `--judge` 없이도 동작. 운영 DB 무영향.

## ③ 피드백 버튼 (👍/👎)

### F-3. 답변 피드백 수집
- **위치**: `static/index.html`(UI), `app/main.py`(엔드포인트), DB 마이그레이션, 신규 `tools/list_negative_feedback.py`
- **작업**:
  1. **UI**: 각 답변 아래 👍/👎 버튼. 👎 클릭 시 선택 입력 코멘트(≤500자) 노출. S-0의 `message_id`를 함께 전송.
  2. **엔드포인트**: `POST /api/feedback` `{ message_id, rating: "up"|"down", comment? }`.
     - 기존 ask와 동일하게 **CSRF(`x-app-token`) 검증**, 모든 SQL **파라미터 바인딩**, `comment` **길이 제한·정제**.
     - **코멘트를 프롬프트로 되먹이지 않는다**(저장 전용).
  3. **DB**: 신규 `feedback` 테이블(`id, message_id FK, rating, comment, created_at`). 같은 메시지 재평가 시 upsert.
  4. **export**: `tools/list_negative_feedback.py` — 👎 답변을 질문·사용된 출처와 함께 출력(②의 평가셋 시드용).
- **완료 기준**: 답변마다 👍/👎가 동작하고 DB에 기록된다. 👎+코멘트가 저장된다. export 스크립트로 👎 목록을 질문·출처와 함께 뽑을 수 있다. 피드백 입력이 어떤 프롬프트에도 주입되지 않는다.

## 권장 실행 순서

1. **S-0**(공통 선행 — `message_id`·`sources` 노출) — ①②③의 토대.
2. **F-1**(출처 표기) — 가장 효과 큼, 환각 방지 효과 덤.
3. **F-2**(평가셋) — 이후 모든 변경의 회귀 안전망. F-1의 출처 표기를 `must_cite`로 검증.
4. **F-3**(피드백) — UI·엔드포인트 독립 작업. F-2 시드를 실데이터로 키움.

(F-2와 F-3은 S-0·F-1 이후 병렬로 진행해도 무방.)

## 결정 필요 항목 (기본값으로 진행, 바꿀 것만 알려주면 됨)

1. 출처 표기 형식: **답변 말미 "근거 문서: …" + UI 패널**(기본). 인라인 각주 형태 원하면 변경.
2. 모델 자기보고 출처 **사후 검증으로 무효 출처 제거**(기본 켬). 끄려면 알려줄 것.
3. 평가셋 **LLM 심판**: 기본 꺼짐(결정적 검사만). 켜두려면 알려줄 것.
4. 피드백 코멘트: **👎에 선택 코멘트(≤500자)**(기본). 평점만 받을지 여부.
5. 👎 export: **스크립트(`tools/`)**(기본) vs 별도 관리 화면 필요 여부.
