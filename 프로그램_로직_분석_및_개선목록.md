# 프로그램 로직 분석 및 개선 목록

> 작성일: 2026-06-09
> 대상: 대학교 학사행정 문서 기반 Q&A 도우미 (FastAPI + Claude CLI)
> 범위: 라우팅·문서선택·PDF·프롬프트·CLI 실행·DB·로깅·프론트엔드 전체 + 운영 로그 대조

---

## 1. 전체 동작 흐름 (한눈에)

```
브라우저(index.html)
  └─ POST /api/ask/stream  { question, session_id, source_id, dev_view }
        │  (CSRF 토큰 x-app-token 검증)
        ▼
  prepare_ask()  ── 동기·블로킹, threadpool에서 실행 ──────────────┐
   1) validate_question (길이/공백)                                │ 여기까지가
   2) 라우팅: source_id > 질문앞 @구분자 > 세션 최근출처 > 되묻기   │ "스트림 열기 전"
   3) _gather_with_links(): 주출처 + 연계출처 문서 수집            │ 단계.
        ├─ local      → build_allowed_files + select_context       │ @규정이면 여기서
        └─ pdf_index  → build_pdf_context                          │ Claude를 한 번
              └─ select_regulations_llm() ← ★ Claude 호출 1회      │ 더 호출함
   4) build_answer_prompt() (시스템규칙 + 요약 + 문서 + 질문)     ─┘
        ▼
  run_claude_stream()  ── claude CLI subprocess, stdin으로 프롬프트
        └─ stream-json delta → SSE delta 로 한 조각씩 전송
        ▼
  persist_ask() → DB 저장 + 운영로그(메타만)
```

### 설계상 잘 된 점
- 파일 접근은 `security.py` 화이트리스트 단일 관문 (root 밖·심볼릭링크·확장자 외 차단)
- SQL은 전부 파라미터 바인딩 (f-string/format 조립 없음)
- 프롬프트 인젝션 격리 (`<<<DOCUMENT>>>` / `<<<QUESTION>>>` 구분자 + 토큰 제거)
- CLI는 shell 미사용·도구 비활성화(`--tools ""`)·격리 cwd·최소 env·타임아웃 강제
- @규정 제목 미매칭 시 Claude 없이 결정적 안내(`direct_answer`) → 불필요한 호출 회피

> 보안 골격은 견고함. 문제는 대부분 **토큰/비용 효율, 중복 호출, 일부 버그**에 집중.

## 2. 경우별 토큰 사용량 분석

전제: 한글 UTF-8 ≈ 3 byte/자, Claude 토큰 ≈ 1자당 0.7~1토큰.
`MAX_TOTAL_CONTEXT_BYTES = 200KB` → 약 **6.8만 자 → 5만~7만 입력 토큰**(상한).

| 경우 | Claude 호출 수 | 입력 토큰(대략) | 비고 |
|---|---|---|---|
| 구분자 없음·세션 출처 없음 | **0** | 0 | 되묻기 JSON만 |
| @규정, 제목 매칭 실패 | **0** | 0 | `direct_answer`로 제목목록 안내 (좋은 설계) |
| @지식(local) 단독 | **1** | ~300(규칙) + 문서(최대 5만~7만) | 문서량이 전부를 좌우 |
| local + 연계출처 N개 | **1** | 위와 동일(200KB 공유 상한) | 단, **파일 스캔 비용**은 N배 |
| **@규정, 매칭 성공** | **2** | ①제목선택(~1~3천) + ②답변(PDF본문 최대 200KB) | ★호출 2회 |

### 핵심 비용 동인 3가지
1. **문서 컨텍스트 200KB가 매 질의마다 통째로 전송됨.** 프롬프트 캐싱이 없어 매번 5만~7만 토큰을 새로 과금. 실제 로그에서 한 질의가 **21개 파일, 30.4초** 소요됨(정상 동작이지만 고비용).
2. **@규정은 질의당 Claude를 2번 호출** (`select_regulations_llm`이 `build_pdf_context` 안에서 `run_claude`를 블로킹 호출). 이 호출이 **스트림이 열리기 전**에 일어나 첫 토큰 지연이 큼.
3. **`MAX_FILES_PER_QUERY=8`이 출처별 상한**이라 연계출처가 있으면 전역 파일 수가 8을 초과(로그의 21개). 의도보다 많은 토큰 유입.

## 3. 다듬어야 할 부분 (전체 목록)

심각도: 🔴높음 / 🟡중간 / ⚪낮음·정리

### A. 토큰 / 비용 / 지연

- 🔴 **A-1. @규정 2회 호출 + 블로킹**
  `app/pdf_source.py:277` 의 `select_regulations_llm()`이 본 답변과 별개로 Claude를 한 번 더 호출. `prepare_ask` 안(스트림 전)이라 첫 토큰 지연이 큼.
  → 옵션: (a) 제목 매칭(`select_regulations`)만 쓰고 LLM 선택 제거, 또는 (b) 키워드 매칭이 후보를 못 좁힐 때만 LLM 호출(현재는 항상 LLM 먼저, 실패 시 키워드 폴백 — 순서가 반대로 되어 비용↑).

- 🔴 **A-2. 전역 문서 수 상한 부재**
  `MAX_FILES_PER_QUERY`가 출처별이라 연계 시 누적됨(`app/router.py:107`). 전역 캡(예: 총 8~10개)을 `_gather_with_links`에 추가 권장.

- 🟡 **A-3. 200KB 상한이 과도**
  5만~7만 토큰/질의. 답변 품질 대비 100KB 또는 그 이하로 낮추면 비용/지연이 절반 수준. 값 재검토 필요(`app/config.py:49`).

- 🟡 **A-4. 연계출처 예산 선착순 고갈**
  첫 출처가 200KB를 다 쓰면 연계출처는 0바이트 기여인데도 스캔은 수행됨(`app/router.py:105-118`). 출처별 최소 보장 예산 또는 균등 분배 고려.

- 🟡 **A-5. 토큰 사용량 미계측**
  stream-json의 `message_delta.usage`(input/output tokens)를 무시 중(`app/claude_runner.py:193`). 이걸 잡아 로그/DB에 남기면 "경우별 실토큰"을 실제로 모니터링 가능. (지금은 추정만 가능)

- ⚪ **A-6. 프롬프트 캐싱 없음**
  CLI `-p` 경로라 적용이 까다롭지만 시스템 규칙+문서를 매번 재전송. 장기적으로 Anthropic SDK 직접 호출로 전환하면 `cache_control`로 문서 블록 캐싱 가능(반복 질의 비용 급감).

### B. 정확성 / 버그

- 🔴 **B-1. 죽은 분류기 코드**
  `app/prompt_builder.py:41-58` 의 `CLASSIFY_RULE`/`build_classify_prompt`는 **어디서도 호출되지 않음**. `category`는 그냥 `src["label"]`. 제거하거나, 의도가 "질문→업무영역 자동 분류"였다면 연결 필요(단, 그럼 호출 1회 추가 = 토큰↑).

- 🟡 **B-2. 스트리밍 중 stderr 미배수(잠재적 교착)**
  `run_claude_stream`은 stdout만 읽고 stderr는 루프 종료 후 읽음(`app/claude_runner.py:184-217`). CLI가 stderr로 많이 쓰면 파이프 버퍼가 차서 블로킹 → 90초 타임아웃까지 멈춤. stderr도 별도 스레드로 드레인 권장.

- 🟡 **B-3. 비스트림 `run_claude` 타임아웃 시 자식 누수**
  `subprocess.run`의 TimeoutExpired는 Windows에서 손자(node) 트리를 못 죽임(`app/claude_runner.py:104`). 스트림판은 `_terminate_tree`로 처리하지만 `run_claude`(=@규정 선택 호출)는 안 함 → 좀비 프로세스 가능.

- ⚪ **B-4. `_gather_with_links` 헤더 바이트 미반영**
  `remaining`에서 `docs`만 빼고 `header` 길이는 제외(`app/router.py:113-116`). 미세하게 예산 초과 가능(영향 작음).

### C. 운영 / 유지보수

- 🟡 **C-1. 콘솔 로그 한글 깨짐**
  `uvicorn.err.log`가 전부 모지바케. 파일핸들러는 utf-8이지만 `StreamHandler`(콘솔)가 Windows cp949로 깨짐(`app/logging_conf.py:49-51`). 콘솔 핸들러 인코딩 지정 또는 콘솔 핸들러 제거.

- 🟡 **C-2. 중복 코드 경로**
  `/api/ask`(비스트림, `handle_ask`)와 `/api/ask/stream`이 거의 같은 로직 2벌. 프론트는 stream만 사용(`static/index.html:647`). 비스트림 경로 제거 또는 공용화로 유지보수 부담 축소.

- ⚪ **C-3. 진단 잔재 파일**
  `data/_diag.py`, `data/_verify_links.py`, `data/_diag_out.txt` 등이 데이터 폴더에 섞여 있음. 정리 권장.

- ⚪ **C-4. CSRF 토큰 재시작마다 변경**
  서버 재시작 시 열려있던 페이지의 POST가 전부 403 → 새로고침 필요(`app/main.py:47`). "수동 재시작" 워크플로와 겹치니 인지 필요(설계상 의도면 OK).

### D. 품질 / UX

- 🟡 **D-1. 스캔 400개 초과 시 조용한 품질 저하**
  텍스트 파일 >400이면 내용 스캔 끄고 파일명 점수만(`app/security.py:179`). 사용자에게 신호 없음 → 대형 폴더에서 답변 근거가 부실해질 수 있음.

- ⚪ **D-2. 매 질의 전체 파일 재읽기**
  `select_context`가 질의마다 모든 텍스트 파일을 디스크에서 읽어 점수화(`app/security.py:186-222`). 캐시(파일 mtime 기준)나 사전 인덱싱 도입 시 지연 감소.

- ⚪ **D-3. `_minimal_env`에 APPDATA/LOCALAPPDATA 없음**
  claude CLI가 Windows 설정 디렉터리를 못 찾을 수 있음(`app/claude_runner.py:60`, `app/config.py:59`). 현재 동작 중이면 OK지만 환경 바뀌면 깨질 소지.

1. **즉시 효과(비용)**: A-1(@규정 호출 순서/제거), A-2(전역 파일 캡), A-3(200KB↓)

## 5. 결정 필요 항목 (정책)

- **@규정 LLM 선택 처리**: 완전 제거 vs 키워드 폴백으로만 vs 현행 유지
- **컨텍스트 상한**: `MAX_TOTAL_CONTEXT_BYTES`(현 200KB)와 전역 파일 수 상한을 얼마로 낮출지
